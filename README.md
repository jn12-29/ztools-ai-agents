# ZTools AI Agents

A ZTools plugin for reusable AI agents, prompt templates, and provider-specific request options.

## Features

- Built-in agents for Chinese-English translation, writing polish, bilingual explanation, image explanation, screenshot diagnosis, VLM OCR, chat, and local native OCR.
- Custom agent CRUD with model selection, prompt templates, streaming mode, and output actions.
- Per-agent request headers and extra OpenAI-compatible request body JSON.
- Optional reasoning display for providers that return `reasoning_content`.
- Optional thinking controls for compatible providers, with model-name based provider detection and manual override.
- Optional image input from files, screenshots, clipboard images, or ZTools image/file commands.
- Per-agent run history for completed input, output, returned reasoning, image attachments, and searchable titles.
- Persisted chat sessions for chat-template agents.
- Plugin-level settings for history retention, title generation, caption model, and image attachment persistence.
- Local native OCR through a configurable RapidOCR-compatible HTTP endpoint.
- Dynamic ZTools commands for custom agents.

## Requirements

- ZTools with plugin AI API support for `headers` and `extraBody` in `window.ztools.ai()`.
- Image input requires a model and ZTools host that pass OpenAI-compatible `image_url` message content through to the provider.
- Local native OCR requires a service compatible with `POST /ocr` multipart field `file`, returning `texts`, `scores`, and `boxes`. The default endpoint is `http://127.0.0.1:8080/ocr`.
- At least one AI model configured in ZTools settings.
- Node.js and pnpm for development.

The plugin uses `window.ztools.ai()` and stores its state with `window.ztools.dbStorage`, so agent settings are included in ZTools plugin data sync.
The current text input and attached image draft are not persisted. Completed runs are saved in per-agent history.

## Development

```bash
pnpm install
pnpm dev
pnpm build
```

During development, import `public/plugin.json` as a ZTools development plugin. The manifest points to `http://127.0.0.1:5187` for dev mode.
Port `5187` is fixed with `--strictPort`; if it is occupied, stop the conflicting process before running `pnpm dev`.

## Runtime Contract

- Built-in feature codes are `manager`, `translate`, `polish`, `explain`, `vision`, `screenshot`, `ocr`, `chat`, and `native-ocr`.
- Custom agents register dynamic feature codes as `agent-<agentId>`.
- Agent configs and completed run history metadata are stored in `window.ztools.dbStorage` under the `state` key.
- Plugin-level settings are stored with the same `state` key.
- Run history is stored per agent and capped by the plugin-level history limit, defaulting to 50 records per agent.
- Chat sessions are stored in the same state payload and capped by the plugin-level history limit per chat-template agent.
- Completed history records include a title. The title can be generated locally, generated by a plugin-level caption model, or disabled.
- AI history-title generation is a separate non-streaming metadata request. It does not use the active Agent prompt or thinking controls, and failure keeps the local fallback title.
- Completed image inputs are stored as ZTools DB attachments when image history persistence is enabled and attachments are available; history records keep only attachment metadata and restore attachments by ID. In non-ZTools mocks, image data may fall back to `state`.
- `headers` are parsed from JSON and converted to string values.
- `extraBody` is merged into the OpenAI-compatible chat completion request, while the host keeps control of `model`, `messages`, and `tools`.
- The Agent streaming toggle reserves `extraBody.stream`: streaming runs send `stream: true`, and non-streaming runs remove any custom `stream` value from `extraBody`.
- When image input is attached, the user message uses OpenAI-compatible content parts: one `text` part plus one `image_url` part per image. Text-only runs keep the original string message content.
- Native OCR runs do not call `window.ztools.ai()`. The renderer asks `public/preload/services.js` to post the selected image to the configured local OCR endpoint, avoiding browser CORS requirements.
- Thinking controls are optional and are merged into `extraBody` after custom JSON. The UI can infer common providers from the selected model ID, but ZTools model metadata does not currently expose provider capability details.
- When an agent follows the ZTools default model, ZTools does not expose that model ID to the plugin. The thinking API falls back to OpenAI / GPT and can be changed manually per agent.
- Thinking controls reserve common thinking keys in `extraBody`; when a thinking control is active, the plugin removes stale thinking keys before applying the selected provider mapping.

## Agent Configuration

Use `{{input}}` in the user template to place selected text or typed input. If the template omits `{{input}}`, the input is appended after the template.
Agent page templates are `standard`, `chat`, and built-in-only `nativeOcr`. Standard agents use the Input / Output / History layout. Chat agents use persisted multi-turn sessions. Native OCR uses an image-first OCR layout and hides AI model, prompt, stream, thinking, and request-body controls.
Choose `跟随 ZTools 默认模型` to use the host default model, or select a concrete model for that agent. The ZTools default model itself is changed in ZTools AI model settings.
New and reset built-in agents default to `是否思考: 关闭`.
Built-in default prompts treat input and visible image text as source content, not instructions; saved built-ins that still match an older default prompt are upgraded on load, while edited prompts are preserved.
New custom agents start with a defensive generic prompt, but edited custom prompts remain fully user-controlled.
Image input can be toggled per agent. `图片细节` maps to OpenAI-compatible image detail values `auto`, `low`, and `high`.
Each run accepts up to 4 images. Supported formats are PNG, JPG, WebP, and GIF. Each image must be 4 MB or smaller.
In the Run view, pressing Enter starts the active agent. Use Shift+Enter when editing the input box and a newline is needed.

Examples:

```json
{"X-Provider-Feature":"on"}
```

```json
{"enable_thinking":true,"temperature":0.2}
```

Thinking controls are separate from reasoning display:

- `显示 reasoning 输出` only controls whether returned `reasoning_content` is shown in the UI.
- `Thinking` controls whether a request should send provider-specific thinking parameters.
- `Thinking Effort` maps to provider-specific strength fields when supported, such as `reasoning_effort`.
- `Thinking Budget` maps to provider-specific budget fields when supported.

Current provider mappings:

- OpenAI / GPT: `reasoning_effort`; the UI narrows effort choices for known model IDs such as `gpt-5.1`, `gpt-5-pro`, GPT-5, and o-series models. Unknown OpenAI-like model IDs keep the full `none`, `minimal`, `low`, `medium`, `high`, and `xhigh` option set. Turning thinking off removes generated thinking fields and does not send `reasoning_effort`.
- Qwen: `enable_thinking` and `thinking_budget`.
- Gemini: `reasoning_effort` or `extra_body.google.thinking_config.thinking_budget`; the plugin sends only one style at a time. Turning thinking off uses `thinking_budget: 0`, which disables thinking on supported Gemini 2.5 models but cannot fully disable thinking on Gemini 3 models.
- DeepSeek: `thinking.type` and `reasoning_effort`; DeepSeek accepts `high` and `max`, while compatible `low` / `medium` map to `high` and `xhigh` maps to `max`.

Provider-specific keys must match the selected model provider's OpenAI-compatible API.

## Packaging

Run `pnpm build`, then import or zip the generated `dist/` directory. Vite copies `public/plugin.json`, `public/logo.png`, and `public/preload/services.js` into `dist/`.
